# FootyCards — Telegram Mini App

Коллекционная карточная игра про футболистов внутри Telegram: паки с игроками
(в том числе тематические коллекции), обмены, пять мини-игр (Memory Sequence,
Card Arena, Футбольный сапёр с уровнями сложности, Пенальти, Штрафной удар),
задания (обычные и премиум), реферальная программа, бесплатный периодический
пак, ежедневные награды, встроенная справка для новичков и полноценная
административная панель.

## Стек

- **Frontend**: React 18 + TypeScript + Vite, React Router, Zustand, TanStack Query,
  Tailwind CSS, Framer Motion, Axios, Telegram Mini Apps bridge (`window.Telegram.WebApp`).
- **Backend**: Python 3.12, FastAPI, SQLAlchemy 2 (async), Alembic, PostgreSQL,
  Pydantic v2, PyJWT (только для админ-сессий), aiogram 3 (бот).
- **Инфраструктура**: Docker, Docker Compose, Nginx (prod), pytest, Vitest.

## Переменные окружения

Скопируйте `.env.example` в `.env` и заполните значения:

```bash
cp .env.example .env
```

| Переменная | Назначение |
|---|---|
| `TELEGRAM_BOT_TOKEN` | Токен бота от @BotFather |
| `TELEGRAM_BOT_USERNAME` | Username бота без `@` (для deep link) |
| `ADMIN_TELEGRAM_IDS` | Telegram ID администраторов через запятую |
| `MINI_APP_URL` | Публичный HTTPS-URL фронтенда (для кнопки бота) |
| `BOT_MODE` | `polling` (dev) или `webhook` (prod) |
| `BOT_WEBHOOK_URL` / `BOT_WEBHOOK_SECRET` / `BOT_WEBHOOK_PORT` | Настройки webhook-режима |
| `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT` | Подключение к PostgreSQL |
| `DATABASE_URL` | Полная строка подключения (`postgresql+asyncpg://...`) |
| `JWT_SECRET` | Секрет для подписи admin-JWT (только админка) |
| `DEV_MODE` | `true` — разрешает вход без реального Telegram (заголовок `X-Dev-Mode: true`). По умолчанию `false`; включайте только для разработки — при `ENVIRONMENT=production` приложение откажется стартовать, если `DEV_MODE=true` |
| `DEV_USER_TELEGRAM_ID` | Telegram ID тестового dev-пользователя |
| `ENVIRONMENT` | `development` / `test` / `production` — влияет на защитную проверку `DEV_MODE` при старте |
| `CORS_ORIGINS` | Разрешённые origin для CORS через запятую |
| `TIMEZONE` | Часовой пояс для расчёта ежедневных наград (например `Europe/Moscow`) |
| `STARTING_BALANCE` | Стартовый бонус монет при регистрации |
| `VITE_API_URL`, `VITE_STATIC_URL` | Базовые URL для фронтенда |
| `NGINX_PORT`, `BACKEND_PORT`, `FRONTEND_PORT` | Порты сервисов |

**Секреты не должны попадать в Git.** `.env` и `.env.local` уже в `.gitignore`.

## Запуск через Docker (рекомендуется)

### Development

```bash
docker compose up --build
```

Поднимет: `postgres`, `backend` (uvicorn --reload, порт 8000), `bot` (polling),
`frontend` (Vite dev-server с HMR, порт 5173).

Применить миграции и заполнить тестовыми данными (в отдельном терминале, после
того как контейнеры поднялись — миграции применяются автоматически при старте
`backend` через `docker-entrypoint.sh`, но seed нужно запустить вручную один раз):

```bash
docker compose exec backend python -m app.seed
```

Frontend: http://localhost:5173
Backend Swagger: http://localhost:8000/api/docs
Админ-панель: http://localhost:5173/admin (нужен Telegram ID из `ADMIN_TELEGRAM_IDS`)

### Production

```bash
docker compose -f docker-compose.prod.yml up --build -d
docker compose -f docker-compose.prod.yml exec backend alembic upgrade head
docker compose -f docker-compose.prod.yml exec backend python -m app.seed   # опционально
```

Поднимет: `postgres`, `backend` (без reload, несколько воркеров), `bot`, `nginx`
(собирает production-бандл фронтенда и отдаёт его + проксирует `/api`, `/static`,
`/bot/webhook` на соответствующие сервисы). Приложение доступно на `http://<host>:${NGINX_PORT}`.

Изображения футболистов и паков подключены как volume (`./backend/static`), поэтому
переживают пересборку контейнеров.

## Запуск без Docker

Требуется локальный PostgreSQL, Python 3.12 и Node.js 20+.

### Backend

```bash
cd backend
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

pip install -r requirements.txt

# Применить миграции
alembic upgrade head

# Заполнить тестовыми данными
python -m app.seed

# Запустить сервер
uvicorn app.main:app --reload --port 8000
```

Backend читает конфигурацию из переменных окружения / `backend/.env` (см. `.env.example`).

### Frontend

```bash
cd frontend
npm install
npm run dev
```

Откроется на http://localhost:5173, использует `VITE_API_URL`/`VITE_STATIC_URL`
из `.env` (по умолчанию — `http://localhost:8000`).

### Telegram-бот

```bash
cd bot
python -m venv .venv
.venv\Scripts\activate   # или source .venv/bin/activate
pip install -r requirements.txt
python bot.py
```

По умолчанию работает в режиме polling (`BOT_MODE=polling`).

## Тесты

### Backend (pytest, 79 тестов)

```bash
cd backend
pytest tests/ -v
```

Использует изолированную in-memory SQLite (не требует поднятого PostgreSQL) —
для проверки бизнес-логики этого достаточно; гонки на уровне `SELECT...FOR
UPDATE` при этом проверяются вручную на реальном PostgreSQL (SQLite не
поддерживает построчные блокировки). Покрывает: проверку Telegram `initData`
(валидная/подделанная подпись/чужой токен), регистрацию и защиту от повторного
стартового бонуса, dev-mode вход, реферальную привязку (начисление рефереру
только после первого платного пака приглашённого, а не при регистрации),
открытие пака (успех/атомарность/недостаточный баланс/гарантированная
редкость/idempotency-key), тематические коллекции карт (CRUD в админке,
исключение неактивных коллекций из ролла паков, фильтр по коллекции,
проброс названия коллекции через вложенные схемы), продажу карточек (включая
предупреждение о последнем экземпляре и блокировку карточек в составе),
обмены (блокировка карточек, принятие/отмена/повторное принятие, запрет
обмена с самим собой), ежедневную награду (получение/повторное получение в
тот же день), бесплатный периодический пак (доступность/повторный запрос
раньше времени/доступность после истечения интервала), задания (обычные и
премиум — назначение 5 слотов, ротация, начисление и повторное начисление
награды, блокировка без подписки на канал, прогресс по рефералам), 5 мини-игр
— Memory Sequence, Card Arena, Футбольный сапёр (в т.ч. уровни сложности и
рост награды за ячейку), Пенальти, Штрафной удар (старт/раунды/начисление и
повторное начисление награды/часовой лимит с `retry_after_seconds`), составы
и матчи (запрет использования чужой карточки, обязательность полного
состава, начисление энергии/наград), защиту административных маршрутов
(отсутствие токена, чужой не-админский токен, валидный админ-токен).

## Настройка BotFather

1. Создайте бота через [@BotFather](https://t.me/BotFather) → `/newbot`, получите токен → в `TELEGRAM_BOT_TOKEN`.
2. `/setmenubutton` → выберите бота → отправьте текст кнопки (например «⚽ Играть») и URL Mini App
   (см. ниже — публичный HTTPS-адрес фронтенда).
3. `/setjoingroups`, `/setprivacy` — по желанию, не обязательны для работы Mini App.
4. Для локальной разработки Telegram требует HTTPS для WebApp URL — нужен
   туннель. Frontend (Vite dev server) проксирует `/api` и `/static` на
   контейнер `backend`, поэтому наружу достаточно пробросить **только порт
   5173** — отдельный туннель/домен для backend не нужен.

   Туннель — через localhost.run (бесплатно, без регистрации):

   ```bash
   ssh -R 80:localhost:5173 nokey@localhost.run
   ```

   Оставьте эту SSH-сессию открытой на всё время работы с Mini App —
   туннель живёт, только пока открыт терминал. Каждый новый запуск ssh
   выдаёт новый случайный адрес вида `https://xxxxx.lhr.life` (на
   бесплатном тарифе домен зарезервировать нельзя).

   Укажите полученный HTTPS-адрес в `/setmenubutton` и в `MINI_APP_URL`, а
   также добавьте его в `CORS_ORIGINS` — удобно делать через
   `docker-compose.override.yml` (см. пример в корне репозитория), не трогая
   `.env`. После правки override-файла — `docker compose up -d`, чтобы
   контейнеры подхватили новые переменные.

## Настройка URL Mini App

- `MINI_APP_URL` в `.env` — адрес, который бот использует в inline-кнопке
  «Открыть FootyCards» (`bot/keyboards.py`).
- Для production это URL вашего nginx (например `https://cards.example.com`).
- Для локальной разработки — туннель к `http://localhost:5173` (см. выше).
  `VITE_API_URL`/`VITE_STATIC_URL` менять не нужно: по умолчанию это
  относительные пути (`/api/v1`, `/static`), которые Vite проксирует на
  backend и внутри туннеля, и при обычном локальном запуске.

## Webhook или polling

- **Polling** (по умолчанию, проще для разработки): `BOT_MODE=polling`. Бот сам
  опрашивает Telegram, не требует публичного адреса.
- **Webhook** (рекомендуется для production): `BOT_MODE=webhook`,
  `BOT_WEBHOOK_URL=https://<ваш-домен>/bot/webhook`, `BOT_WEBHOOK_SECRET=<случайная строка>`.
  В `docker-compose.prod.yml` nginx уже проксирует `/bot/webhook` на сервис `bot`
  (порт `BOT_WEBHOOK_PORT`, по умолчанию 8081).

## Production-развёртывание (кратко)

1. Настройте DNS/сервер, установите Docker + Docker Compose.
2. Склонируйте репозиторий, создайте `.env` из `.env.example` с реальными
   секретами (`JWT_SECRET`, `TELEGRAM_BOT_TOKEN`, `DB_PASSWORD` и т.д.), `DEV_MODE=false`.
3. Настройте HTTPS перед nginx (например через внешний reverse-proxy/Let's
   Encrypt-терминатор — конфигурация в этом репозитории поднимает только HTTP на 80 порту).
4. `docker compose -f docker-compose.prod.yml up --build -d`.
5. `docker compose -f docker-compose.prod.yml exec backend alembic upgrade head`.
6. Заполните продовые данные о футболистах через админ-панель (`/admin/players`,
   создание вручную или импорт CSV) — тестовый `app.seed` предназначен для demo/dev.
7. В BotFather укажите production `MINI_APP_URL`, переключите бота на webhook.

## Реализованные функции

- Безопасная Telegram-авторизация (проверка HMAC-подписи `initData` на
  сервере, автосоздание пользователя, стартовый бонус выдаётся один раз,
  dev-режим только при явном включении, недоступен в production).
- Игровая валюта: стартовый бонус, ежедневная награда, 5 мини-игр, матчи,
  задания, продажа карточек, админ-корректировка — вся логика на backend,
  атомарные транзакции, журнал `coin_transactions`. Экономика (стоимость
  паков, награды за игры/задания, интервал бесплатного пака) сбалансирована
  и настраивается через `GameConfig` в админке.
- Футбольные карточки: 44 тестовых футболиста всех 4 редкостей, 12 позиций,
  уникальные экземпляры с серийным номером, блокировки (в составе/в обмене/
  админом), локальные сгенерированные изображения (без внешних загрузок).
  Часть футболистов может входить в тематические **коллекции карт**
  (управляются в админке, название коллекции отображается на карточке, в
  деталях, при открытии пака и в обменах); карточки из неактивной коллекции
  исключаются из ролла паков.
- 3 пака (Basic/Premium/Elite) с настраиваемыми вероятностями редкости,
  гарантированной минимальной редкостью, лимитами покупок и периодом продажи;
  открытие — атомарная транзакция с idempotency-key и защитой от гонки
  повторных запросов (проверено тестами и вручную на реальном PostgreSQL).
  Плюс отдельный **бесплатный пак** по таймеру (настраиваемый интервал).
- Поэтапная анимация открытия пака на Framer Motion (позиция → редкость →
  страна → клуб → силуэт → полное раскрытие), эффект свечения по редкости,
  ускорение по тапу, кнопка «Пропустить всё», итоговый экран с отметками
  «Новая»/дубликаты и названием коллекции для тематических карт.
- Коллекция: фильтры (редкость/страна/клуб/позиция/рейтинг/коллекция карт),
  поиск, сортировка, сетка, детальная карточка, быстрая и массовая продажа с
  подтверждением и предупреждением о последнем экземпляре.
- 5 мини-игр, у каждой лимит 3 попытки в час (с понятным сообщением об
  оставшемся времени, а не сырой ошибкой) и дневной лимит наградных попыток:
  - **Memory Sequence** — серверная сессия и раунды, длина растёт с каждым
    уровнем, рекорд, таблица лидеров.
  - **Card Arena** — состав 4-3-3, расчёт силы команды на backend (рейтинг +
    позиция + редкость + химия клуба/страны + случайность), матчи против
    бота или реального состава другого игрока, хронология событий, дневная
    энергия, статистика, рейтинг Arena, таблица лидеров, история матчей.
  - **Футбольный сапёр** — уровни сложности (1-4 бомбы на поле): чем больше
    бомб, тем выше награда за открытие безопасной ячейки, но и риск выше.
  - **Пенальти** — серия ударов против бота, шанс промаха зависит от
    рейтинга выбранной карточки.
  - **Штрафной удар** — таймингова игра на движущейся шкале; попадание в
    зону оценивается сервером по собственному замеру времени (не по
    значению, присланному клиентом), чтобы исключить читерство с таймингом.
- Задания: обычные (ротация 5 слотов из общего пула, обновляются при
  выполнении) и премиум (требуют подписки на Telegram-канал), начисление
  прогресса по серверным метрикам (открытые паки, уникальные футболисты,
  завершённые обмены, приглашённые друзья, сила состава в матче).
- Реферальная программа: пригласительная ссылка через бота, награда
  рефереру начисляется только после того, как приглашённый друг сделает
  первый платный (не бесплатный) пак — не мгновенно при регистрации, чтобы
  нельзя было фармить награду одноразовыми аккаунтами.
- Обмены: поиск игрока, предложение карточек/монет, блокировка карточек на
  время сделки (с построчной блокировкой БД, чтобы одну карточку нельзя было
  продать двум разным людям одновременно), повторная проверка при принятии,
  авто-истечение через 24 часа, уведомления через Telegram-бота.
- Ежедневная награда на 7 дней (монеты/бесплатный пак/случайная карточка),
  проверка даты на backend с настраиваемым часовым поясом.
- Профиль (приватный/публичный), рейтинг, история транзакций (только у
  владельца/админа).
- Онбординг: кнопка «Помощь» в шапке мини-аппа с кратким FAQ по разделам,
  точечные подсказки на экранах (часовой лимит игр, режим сложности сапёра
  и т.д.) — чтобы новый игрок не терялся в интерфейсе.
- Административная панель: дашборд с графиками, пользователи (баланс, бан,
  выдача карточек, сброс лимитов, блокировка игровых наград), футболисты
  (CRUD, загрузка/удаление изображений, CSV импорт/экспорт, запрет удаления
  при наличии выданных карточек), тематические коллекции карт (CRUD,
  переключение активности), паки (CRUD, предпросмотр вероятностей), задания
  (CRUD, рассылка уведомления о новом премиум-задании), обмены (принудительная
  отмена), игры (настройка наград/лимитов/сложности сапёра, подозрительные
  результаты), журнал административных действий.
- Telegram-бот: `/start`, `/profile`, `/help`, deep link приглашения,
  автодоставка уведомлений об обменах, напоминание о ежедневной награде,
  админ-команды (`/admin`, `/give_coins`, `/give_card`, `/ban`, `/unban`,
  `/stats`, `/announce_pack`) с проверкой Telegram ID.
- Безопасность: HMAC-проверка initData, dev-режим не активируется в
  production (жёсткая проверка при старте), серверная проверка всех игровых
  операций (сервер пересчитывает результат сам, а не доверяет данным от
  клиента), защита от отрицательного баланса, построчные блокировки БД
  (`SELECT...FOR UPDATE`) для всех операций с монетами/картами/обменами —
  включая повторную проверку состояния уже ПОСЛЕ получения блокировки, чтобы
  параллельные запросы не проходили одну и ту же проверку дважды,
  idempotency-key для открытия пака, rate limiting критичных операций,
  валидация загружаемых файлов (тип/размер/безопасное имя), единый формат
  ошибок без внутренних stack trace, CORS по белому списку.

## Команды запуска (шпаргалка)

```bash
# Docker dev
docker compose up --build
docker compose exec backend python -m app.seed

# Docker prod
docker compose -f docker-compose.prod.yml up --build -d
docker compose -f docker-compose.prod.yml exec backend alembic upgrade head

# Без Docker
cd backend && alembic upgrade head && python -m app.seed && uvicorn app.main:app --reload
cd frontend && npm install && npm run dev
cd bot && python bot.py

# Тесты
cd backend && pytest tests/ -v
cd frontend && npm run test
```

## Известные ограничения

- Массовая рассылка `/announce_pack` и напоминание о ежедневной награде из бота
  используют `bot.send_message` в цикле по всем пользователям — для очень
  большой базы пользователей это стоит заменить на очередь с ограничением
  скорости (Telegram лимитирует ~30 сообщений/сек).
- Rate limiting реализован in-memory (на процесс) — для горизонтального
  масштабирования backend на несколько инстансов нужен внешний стор (Redis).
- Прогресс заданий начисляется по ограниченному набору серверных метрик
  (`packs_opened`, `unique_players`, `trades_completed`, `referrals_count`,
  минимальный рейтинг состава в матче) через `task_service.evaluate_metric_progress`/
  `evaluate_match_min_rating`; добавление новой метрики требует явного вызова
  в нужном сервисе.
- Известные, но не исправленные гонки (низкий приоритет, не дают бесплатных
  монет): лимит покупок конкретного пака (`purchase_limit_per_user`) можно
  ненадолго превысить параллельными запросами; внутри одной ещё не завершённой
  игровой сессии (`submit_round`/`reveal_cell`/`resolve_kick`) параллельные
  запросы могут исказить промежуточный счёт до итогового начисления награды.
- Изображения футболистов — процедурно сгенерированные заглушки (цветной
  градиент по редкости + инициалы), а не реальные фотографии, как и требовалось
  в задании («не скачивай фотографии из интернета»); имена футболистов —
  вымышленные, чтобы не использовать чужие права на изображение/имя.
- CSV-импорт футболистов сопоставляет строки по `display_name` (upsert) — при
  массовом импорте с опечатками в имени возможны дубликаты.
- `npm audit` может показывать уязвимости в dev-зависимостях (обычная ситуация
  для Vite/Vitest-тулчейна) — не стоит бездумно гонять `npm audit fix --force`,
  это может подтянуть breaking-changes.
